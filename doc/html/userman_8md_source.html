41 ## I love it when a plan comes together
42 
43 Before anything else, you need a testing plan. This basically declares
44 how many tests your script is going to run to protect against premature
45 failure.
46 
47 - **plan()**
48 
49  void plan(int number_of_tests);
50  void plan(skip_all, const std::string& reason="");
51  void plan(no_plan);
52 
53  The function `plan` is used to indicate the plan of your test run. Usually you
54  will just give it the number of tests as argument.
55 
56  Alternatively, you can give it the `skip_all` or `no_plan` constants as
57  arguments. The first means you will not run the tests at all, the second means
58  you will run an unknown number of tests (the latter is not recommended).
59 
60 - **done\_testing()**
61 
62  void done_testing();
63  void done_testing(int number_of_tests);
64 
65  If you don't know how many tests you're going to run, you can issue the plan
66  when you're done running tests.
67 
68  number\_of\_tests is the same as plan(), it's the number of tests you expected to
69  run. You can omit this, in which case the number of tests you ran doesn't
70  matter, just the fact that your tests ran to conclusion.
71 
72  This is safer than and replaces the "no\_plan" plan.

74 ## Test names
75 
76 By convention, each test is assigned a number in order. This is
77 largely done automatically for you. However, it's often very useful to
78 assign a name to each test. Which would you rather see:
79 
80  ok 4
81  not ok 5
82  ok 6
83 
84 or
85 
86  ok 4 - basic multi-variable
87  not ok 5 - simple exponential
88  ok 6 - force == mass * acceleration
89 
90 The later gives you some idea of what failed. It also makes it easier
91 to find the test in your script, simply search for "simple
92 exponential".
93 
94 All test functions take a name argument. It's optional, but highly
95 suggested that you use it.

97 ## I'm ok, you're not ok.
98 
99 The basic purpose of this module is to print out either "ok #" or "not
100 ok #" depending on if a given test succeeded or failed. Everything
101 else is just gravy.
102 
103 All of the following print "ok" or "not ok" depending on if the test
104 succeeded or failed. They all also return true or false,
105 respectively.
106 
107 - **ok()**
108 
109  bool ok(bool condition, const std::string& test_name = "");
110 
111  `ok` is the basic test expression in TAP. It simply evaluates any expression,
112  for example, _got == expected_, taking a true value to mean that the test
113  passed and a false value to mean that the test failed.
114 
115  `test_name` is a very short description of the test that will be printed out.
116  It makes it very easy to find a test in your script when it fails and gives
117  others an idea of your intentions. `test_name` is optional, but we very
118  strongly encourage its use.
119 
120 - **is()**
121 - **isnt()**
122 
123  template<typename T, typename U> bool is(const T& got, const U& expected, std::string& test_name = "");
124  template<typename T, typename U> bool isnt(const T& got, const U& expected, std::string& test_name = "");
125 
126  Similar to ok(), is() and isnt() compare their two arguments
127  with `==` and `!=` respectively and use the result of that to
128  determine if the test succeeded or failed. So these:
129 
130  # Is the ultimate answer 42?
131  is( ultimate_answer(), 42, "Meaning of Life" );
132 
133  # foo isn't empty
134  isnt( foo, "", "Got some foo" );
135 
136  are similar to these:
137 
138  ok( ultimate_answer() == 42, "Meaning of Life" );
139  ok( foo != "", "Got some foo" );
140 
141  (Mnemonic: "This is that." "This isn't that.")
142 
143  So why use these? They produce better diagnostics on failure. ok() cannot know
144  what you are testing for (beyond the name), but is() and isnt() know what the
145  test was and why it failed. For example this test:
146 
147  std::string foo("waffle"), bar("yarblokos");
148  is( foo, bar, 'Is foo the same as bar?' );
149 
150  Will produce something like this:
151 
152  not ok 17 - Is foo the same as bar?
153  # Failed test 'Is foo the same as bar?'
154  # got: 'waffle'
155  # expected: 'yarblokos'
156 
157 - **pass()**
158 - **fail()**
159 
160  bool pass(const std::string& test_name = "");
161  bool fail(const std::string& test_name = "");
162 
163  Sometimes you just want to say that the tests have passed. Usually
164  the case is you've got some complicated condition that is difficult to
165  wedge into an ok(). In this case, you can simply use pass() (to
166  declare the test ok) or fail (for not ok). They are synonyms for
167  ok(true, test\_name) and ok(false, test\_name).
168 
169  Use these very, very, very sparingly.

171 ## Conditional testing
172 
173 - **skip()**
174 
175  void skip(int number, const std::string& reason = "");
176 
177  `skip` tells the TAP harness that you're skipping a _number_ of tests for the
178  given _reason_. Note that you have to do the skipping yourself.
179 
180 - **TODO**
181 
182  {
183  todo_guard why;
184  TODO="why"
185  my_tests_here ...
186  }
187 
188  `TODO` is a global string variable that tells TAP harness the reason
189  the current test is expected to fail. You set TODO before a block of
190  tests that you expect to fail and then unset it afterwards. When TODO
191  is the empty string, then the harness considers that there is no
192  reason for the test to fail. However, when TODO is non-empty, any
193  failing test is not counted against the test suite and any succeeding
194  test is reported as an unexpected success.
195 
196  The nice part about todo tests, as opposed to simply commenting out a
197  block of tests, is it's like having a programmatic todo list. You know
198  how much work is left to be done, you're aware of what bugs there are,
199  and you'll know immediately when they're fixed.
200 
201  Note that TODO manipulates a global variable. Thus, you should be
202  careful to set it to "" before going to another section of the
203  program. An easy mistake to make is to have a failing section of code
204  that throws an exception, taking you out of the current scope without
205  resetting TODO. To make it easier to deal with this in a thread-safe
206  manner, the todo\_guard class is provided. Objects of this class will
207  reset TODO when they fall out of scope.

209 ## Diagnostics
210 
211 If you pick the right test function, you'll usually get a good idea of
212 what went wrong when it failed. But sometimes it doesn't work out
213 that way. So here we have ways for you to write your own diagnostic
214 messages which are safer than just `print STDERR`.
215 
216 - **diag**
217 
218  diag(diagnostic_message...);
219 
220  Prints a diagnostic message which is guaranteed not to interfere with
221  test output. The arguments are simply concatenated together.
222 
223  Returns false, so as to preserve failure.
224 
225  Handy for this sort of thing:
226 
227  ok( has_user("foo"), "There's a foo user" ) or
228  diag("Since there's no foo, check that /etc/bar is set up right");
229 
230  which would produce:
231 
232  not ok 42 - There's a foo user
233  # Failed test 'There's a foo user'
234  # Since there's no foo, check that /etc/bar is set up right.
235 
236  You might remember `ok() or diag()` with the mnemonic `open() or
237  die()`.
238 
239  **NOTE** The exact formatting of the diagnostic output is still
240  changing, but it is guaranteed that whatever you throw at it it won't
241  interfere with the test.
242 
243 - **note**
244 
245  note(diagnostic_message...);
246 
247  Like diag(), except the message will not be seen when the test is run
248  in a harness. It will only be visible in the verbose TAP stream.
249 
250  Handy for putting in notes which might be useful for debugging, but
251  don't indicate a problem.
252 
253  note("Tempfile is ", tempfile);
254 
255  `diag` simply catenates its arguments to the error output, while `note`
256  prints diagnostics to the TAP stream.
257 
258 - **set\_output()**
259 - **set\_error()**
260 
261  void set_output(std::ofstream& new_output);
262  void set_error(std::ofstream& new_error);
263 
264  These set the filehandle of the TAP stream and the error stream. They default
265  to `std::cout` and `std::cerr`, respectively. These can only be set before any
266  output is written to them.

268 ## Ending a test run
269 
270 - **exit\_status()**
271 
272  If all your tests passed, Test::Builder will exit with zero (which is
273  normal). If anything failed it will exit with how many failed. If
274  you run less (or more) tests than you planned, the missing (or extras)
275  will be considered failures. If the test died, even after having
276  successfully completed all its tests, it will still be considered a failure
277  and will exit with 255.
278 
279  So the exit codes are...
280 
281  0 all tests successful
282  255 test died or all passed but wrong # of tests run
283  any other number how many failed (including missing or extras)
284 
285  If you fail more than 254 tests, it will be reported as 254.
286 
287 - **bail\_out()**
288 
289  int exit_status();
290  void bail_out(const std::string& reason);
291 
292  **bail\_out** terminates the current test program with exit code 255, indicating
293  to the test harness that all subsequent testing should halt. Typically this
294  is used to indicate that testing cannot continue at all.
